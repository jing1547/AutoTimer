## 1. 재생 창 Z-order 강제 끌어올림

### 배경
- 테스트 재생(트레이 메뉴 → 테스트 재생)은 JW 라이브러리 서브 창(Topmost) 위로 잘 떴으나, 실제 스케줄 자동 재생 시 **JW 서브 창 뒤에 숨어서** 재생됨.
- 백그라운드 프로세스가 만드는 창은 Windows의 `SetForegroundWindow` 포커스 도둑 방지 정책 때문에 Z-order 상단 진입이 제한됨.

### 원인
1. 이전 코드: `Topmost = true`가 `LockWindow` 설정에 묶여 조건부로만 적용
2. WPF `Topmost`만으로는 이미 Topmost인 타 프로세스 창 위로 올라가지 못함
3. 트레이 상태에서 스케줄 트리거 시 AutoTimer 프로세스가 foreground가 아니라서 `SetForegroundWindow` 호출이 거부됨
4. 설정창이 비활성 상태로 열려 있으면 WPF 프로세스가 foreground가 아니어서 동일한 제약

### 해결
**`VideoWindow.xaml.cs` — `ForceToTop()` 추가**
- `SPI_SETFOREGROUNDLOCKTIMEOUT`를 0으로 임시 변경하여 포커스 락 해제
- 현재 foreground 스레드에 `AttachThreadInput`으로 스레드 입력 큐를 붙여 `SetForegroundWindow` 제약 우회
- `ShowWindow(SW_SHOW)` → `SetWindowPos(HWND_TOPMOST)` → `BringWindowToTop` → `SetForegroundWindow` → `Activate` 순차 호출
- `finally`에서 AttachThreadInput 해제 + 타임아웃 원복
- `Topmost = true`를 `LockWindow`와 무관하게 **항상** 설정

**`App.xaml.cs` — `PreWindowActivation.EnsureProcessForeground()` 추가**
- `StartPlayback` 전에 호출
- AutoTimer가 소유한 보이는 창(주로 설정창)을 찾아 AttachThreadInput 우회로 foreground로 끌어올림
- 이로써 이어서 생성되는 VideoWindow의 `ForceToTop`이 제약 없이 성공
- 설정창이 떠있는 상태에서 비활성일 때의 Z-order 실패 해결

**`NativeMethods.cs` — P/Invoke 추가**
- `SetWindowPos`, `SetForegroundWindow`, `ShowWindow`, `GetForegroundWindow`, `GetWindowThreadProcessId`, `GetCurrentThreadId`, `AttachThreadInput`, `BringWindowToTop`, `SystemParametersInfo`
- 관련 상수: `HWND_TOPMOST`, `SWP_NOMOVE`, `SWP_NOSIZE`, `SWP_SHOWWINDOW`, `SWP_NOACTIVATE`, `SW_SHOW`, `SW_RESTORE`, `SPI_GETFOREGROUNDLOCKTIMEOUT`, `SPI_SETFOREGROUNDLOCKTIMEOUT`, `SPIF_SENDCHANGE`

### 결과
- 전체화면 앱/Topmost 타 프로세스 창 앞으로 정상 배치
- 트레이 상태, 설정창 활성/비활성, JW 라이브러리 서브 창 등 모든 케이스에서 재생 창이 최상위

## 3. PrePlaybackNotification (1분 전 안내 창) 버튼 개선

### 배경
기존에는 `재생 취소` 버튼만 존재. 사용자가 확인 버튼이 없으면 "이 창을 어떻게 없애지?" 혼란. 또한 취소 버튼이 재확인 없이 즉시 동작해 함정처럼 느껴짐.

### 변경
**`PrePlaybackNotification.xaml`**
- `확인` 버튼 추가 (`BtnOk`), `재생 취소` 버튼과 나란히 배치

**`PrePlaybackNotification.xaml.cs`**
- `BtnOk_Click`: 타이머 정지 + 창만 닫음. 재생은 스케줄러의 `ScheduleTriggered`가 정시에 자동 실행.
- `BtnCancel_Click`: `CustomDialog.ShowYesNo`로 재확인.
  - 예 → 취소 확정, 창 닫음
  - 아니오 → 타이머 재개, 창 유지
- 버튼 스타일: 확인(강조색), 취소(일반색)

**`App.xaml.cs` — `OnPreNotification` 핸들러**
- 기존: `cancelled=false`면 `StartPlayback`을 직접 호출 (카운트다운 완료 시 즉시 재생)
- 변경: `cancelled=false`여도 아무 동작 안 함. 재생은 `ScheduleTriggered`가 정시에 트리거
- `cancelled=true`면 `_playbackSuppressed=true`로 이 스케줄의 `ScheduleTriggered` 1회 억제

### 결과
- 확인 버튼으로 안내 창만 치울 수 있어 혼란 해소
- 취소 버튼은 재확인 단계를 거쳐 실수 방지
- 재생 시점이 스케줄 시각에 정확히 일치 (사용자 확인 타이밍에 무관)

## 5. 스케줄 시간 포맷 HH:MM → HH:MM:SS로 확장

### 변경 파일

**`Models/AppSettings.cs`**
- `WeeklySchedule.Time`, `OneTimeSchedule.Time` 기본값 `"00:00"` → `"00:00:00"`

**`Services/SchedulerService.cs`**
- `_lastCheckedMinute`/`_lastPreCheckedMinute` → `_lastCheckedSecond`/`_lastPreCheckedSecond`
- 분 단위 체크 → **초 단위 체크**: `currentSecond = (now.Hour * 60 + now.Minute) * 60 + now.Second`
- `TryParseTime(string time, out int hour, out int minute, out int second)`: 세 번째 out 파라미터 추가, 두 컴포넌트(`"HH:mm"`)만 있는 legacy 값도 받아 `second=0`으로 처리
- `NormalizeTime(string)`: `"HH:mm"` 또는 `"HH:mm:ss"` 모두 `"HH:mm:ss"`로 정규화하는 헬퍼
- `CheckTrigger`의 문자열 비교를 `NormalizeTime` 기반으로 변경
- `CheckPreNotification`을 **문자열 매칭 → DateTime 범위 비교**로 재작성:
  - 각 스케줄의 정확한 `scheduleTime` 계산 (date + h:m:s)
  - 현재 시각이 `[scheduleTime - 1분, scheduleTime)` 구간에 들어오면 알림 발송

**`Controls/TimePicker.xaml.cs`**
- `Time` DependencyProperty 기본값 `"00:00"` → `"00:00:00"`
- `UpdateDisplay`: legacy `"HH:mm"` 값을 자동으로 `":00"`을 붙여 표시 (마이그레이션 불필요)

**`SettingsWindow.xaml`** — 스케줄 그리드 컬럼 재조정
- 주간 스케줄: `56/72/68/*/28` → `56/64/92/*/28` (요일 콤보 축소, TimePicker 확장)
- 일회성 스케줄: `112/68/*/28` → `108/92/*/28` (날짜 살짝 축소, TimePicker 확장)

**`SettingsWindow.xaml` — TimePicker 팝업**
- 초 StackPanel 추가 (`TxtPickerSec` + 상하 버튼)
- 각 컬럼 하단에 `"시"` / `"분"` / `"초"` 라벨 TextBlock 추가

**`SettingsWindow.xaml.cs` — 피커 로직**
- `_pickerSec` 필드 추가
- `OnSecUp`/`OnSecDown` 이벤트 핸들러 추가
- `OnTimePickerRequested`에서 초 파싱
- `UpdateTimePickerDisplay`에서 `TxtPickerSec` 갱신
- `OnTimeOk`: `$"{h:D2}:{m:D2}"` → `$"{h:D2}:{m:D2}:{s:D2}"`

### 마이그레이션
- 기존 설정 JSON의 `"HH:mm"` 값은 `TryParseTime`과 `UpdateDisplay`에서 자동으로 `":00"`이 붙은 것처럼 취급
- 저장 파일 변경 없이 호환

## 6. 겹치는 스케줄 예외 처리

### 배경
영상 길이가 5분인데 그 5분 구간 안에 여러 개의 스케줄이 겹쳐 있으면 중복 재생 시도 가능.

### 정책
**"먼저 시작된 영상 기준. 이후 겹치는 스케줄은 무시."**

### 확인/변경
- **자동 재생 (`OnScheduleTriggered`)**: 이미 `App.xaml.cs:100`에 `_activeVideo is not null && !_activeVideo.IsTestPlay`면 return. 기존 로직이 이미 정책에 맞음.
- **사전 알림 (`OnPreNotification`)**: 이미 `App.xaml.cs:60`에 `_activeVideo is not null`면 팝업 자체를 띄우지 않음. 기존 로직 OK.
- **새로고침 (`FindCurrentSchedule`)**: 기존에는 `c.startTime > best.Value.startTime`로 **가장 최근** 시작된 스케줄을 고름 → `<`로 변경하여 **가장 먼저 시작된** 스케줄 선택.

### 결과
새로고침을 눌렀을 때 겹치는 스케줄이 있어도 맨 처음 시작된 스케줄 기준으로 seek 동기화. 중복 재생·뒤엎기 방지.
